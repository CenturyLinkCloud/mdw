Introduction on Developing Activity Implementor

The following steps are needed to define an activity implementor:
  • Create an implementor Java class
  • Using Designer, add the implementor to the tool panel

Create Implementor Class

The main requirement for an activity implementor class that extends com.centurylink.mdw.workflow.activity.DefaultActivityImpl, or one of its subclasses, out-of-box or user defined. It is more common to extends an out-of-box implementor class, and for adapter, task, event wait, timer wait, synchronization activities, it is almost impossible not to start from the out-of-box activities. The rest of this document covers how to implement activity implementor from the base activity and common tasks for new implementors, please refer to corresponding documentation when extending from an out-of-box activity implementor.

There is really only one method that must be implemented by an activity implementor:

public void execute() throws ActivityException
The code logic to execute this activity. The default implementation does nothing.

The following methods are highly recommended to be overriden. They are used by the designer to obtain descriptions of the implementor:

public String getLabel()
Return the label of the activity implementor displayed in the tool pane of the designer. Any string is acceptable but you may want to ensure the name is short and unique. If you do not override this method, the default method returns Dummy Activity.
public String getIconName()
Return the icon name of the activity implementor. The icon name must be one of the following:
  • an image stored in the database as resource (with language IMAGE_JPEG, IMAGE_GIF or IMAGE_PNG). The name specified here must be the resource name, followed by .jpg, .gif or .png.
  • an image file name packaged together with designer (this is no longer recommended, so we are not listing available image file names here);
  • one of the following internal shapes: shape:start, shape:start, shape:recvmsg, shape:sendmsg, shape:task, shape:rule, shape:subproc, shape:invoke, shape:decision, shape:activity
If you do not override this method, the default method returns shape:activity. For consistent look-and-feel, we recommend that your icon size be 24 by 24 or slightly smaller.
Additionally, you can implement the following trace annotation
@Tracked(LogLevel.TRACE)
You can apply this annotation to your implementor class to automatically track timings for activity lifecycle stages. With the annotation applied, timings will be logged as follows if logging is turned on at the specified level for your implementation class.

If the implementor needs a constructor, it must have one taking no argument, and it should always invoke the base constructor first.

In a later section, we summarize all methods that are defined in the base class and can be used in your implementor. You should only use these methods, and Eclipse IDE makes it easy to find out what methods are available.

Additional notes:

  • Do not reference to variables by their names directly – use attributes to get their names. In this way, your implementor can easily be reused in a different place, as variable names can be configured at design time.
  • Use getAttributeValueSmart instead of getAttributeValue, so that attribute value can resolve binding expressions with dynamic runtime values in addition to static text.
  • If you extend some subclasses, such as AdaterActivityBase, you may need to override some of the specific methods used by base implementation of execute(), instead. For activities implementing SuspendibleActivity, you will need to implement resume(), or override it from an existing implementor implementing the activity interface.
  • Use setReturnCode() to set return code (default null). You can also set it to ABORT:, CORRECT:, DELAY:, ABORT:mycode, etc. Do not set it to ERROR: – throw exception instead.

Attribute Descriptor

The Attribute Definition Tab for an activity implementor in the designer needs to be customized to allow defining attributes specific to the implementor. The customization is achieved through an attribute descriptor, which is an XML string stored as an attribute of the implementor. As an example, the descriptor for ScriptEvaluator activity implementor looks like: 
    <PAGELET>
        <RULE NAME='Expression'/>
        <HYPERLINK URL='../doc/index.html'>
            Click here for script language document</HYPERLINK>
    </PAGELET>
The XML descriptors follow a simple syntax:
  • The root element is always PAGELET
  • Each first level child element defines a widget in the dialog, most typically representing an attribute required by the implementor.
  • Each first level child element has a set of attributes for common things like the size and location of the widget
The following table lists all supported XML attributes of widgets.
XML AttributeDescription
NAMEName of the activity attribute. This is required if the widget describes an activity attribute.
LABELLabel shown on the dialog. If this is missing, then NAME is used as the label
LXX coordinate of the label. If this is not present, the location will be automatically computed.
LYY coordinate of the label. If this is not present, the location will be automatically computed.
LWWidth of the label. The default is 160 (pixels) if this is not present
VXX coordinate of the value area. If this is not present, the location will be automatically computed.
VYY coordinate of the value area. If this is not present, the location will be automatically computed.
VWWidth of the value area. If this is not present, the width of the input area takes a default value specific to each widget type.
VHHeight of the value area. If this is not present, the height of the input area takes a default value specific to each widget type.

In the following we list all supported widgets.

TEXT
A general text input field. There is no nested element. An additional attribute SIZE is used to specify the maximum number of characters allowed in the text input field
RULE
A text input area for scripting languages. There is no nested element.
LIST
This widget allows to set attribute is a list of values, separated by "#". The attribute DELIMITER can be used to change the delimiter.
SELECT
This displays a group of radio buttons for the user to select from one value out of a list. The list of candidate values are specified by child elements OPTION, one for each option.
NOTE
This simply generates some static text, typically for instructions. The actual text to display is included as the contents of the tag.
HYPERLINK
This generates a hyper link to a URL. The URL is specified in the attribute "URL" and the content of the tag will be displayed as a hyper link.
MAPPING
This is used to specify parameter mappings for process invocation activities. It displays a table of formal parameters needed by the subprocess and allows user to specify actual parameters. No nested elements and no additional attributes are supported by this widget.

Configure the Implementor in Designer

Your implementors must be configured in the designer to be used. The following information needs to be provided during configuration.
  • Activity Implementor Class. This the class name of your implementor. When the designer is running in on-line mode (through server), it will validate the accessibility of the class in the server during save, so you need to ensure the class is deployed prior to configuring the implementor. The designer does not validate the class when it is not running in server mode.
  • Activity Implementor Label. This is the label displayed besides the icon in the implementor tool pane. See getLabel() method above for additional information.
  • Activity Type. This is a drop down list of base activity implementor class names. This information is used by the designer to provide activity type specific features and there is no runtime implication. The following describes the function of each type.
    • Activity. This is the basic type and there is no special functionality provided by the designer.
    • AdaptorActivity. The run time information will display external messages, and the simulation tab allows stubbing.
    • EventWaitActivity. The run time information will display external messages.
    • StartActivity.
    • FinishActivity.
    • SynchronizationActivity. Allows to configure synchronization in designer
    • InvokeProcessActivity. Allows to configure parameter passing, and subprocess names, etc.
    • TaskActivity. A manual task activity.
    • ScriptActivity. This allows the designer running inside Eclipse to pop up editor panel.
  • Icon name. See description for getIconName() method above for additional information. With MDW 5 designer, a drop down list with all available icon names is shown for selection.
  • Attribute descriptor. This has been described above.

With MDW 5 Designer, you only need to provide the implementor class name, and the rest of the information can be automatically loaded by pressing the button "Load from Class" while the server is running and the class is already accessible.

API For Implementing Implementor

he base class DefaultActivityImpl contains methods (API) for assisting your implementations. Please refer to its java doc (following above hyperlink, or through Eclipse IDE) for detailed description of the methods. The following list summarizes common API methods.
  • Attribute accessing methods:
    • getAttributeValue: get attribute value of the activity by attribute name
    • getAttributeValueAsVariableName: assuming the attribute specifies a variable (either an identifier, or a dollar sign followed by an identifier), return the variable name
    • getAttributeValueSmart: same as getAttributeValue but allows binding expression syntax
    • getAttributes: return a list of all attributes
  • Variable and document accessing and updating:
    • getParameterValue: get variable instance (a.k.a parameter) value. Despite the method name, this is used for all variables, not just parameters of the process.
    • setParameterValue: set variable instance (parameter) value
    • getParameterId: get the variable instance ID
    • getParameterType: get the type of the variable
    • getParameters: get all variable instances for the process instance
    • setParameterValueAsDocument: update/create a document and set the value of the variable instance to a document reference
    • getDocument: return the document referenced by a document reference
    • getDocumentForUpdate: same as getDocument except putting a write lock on the document
    • getDocumentContent: return the content of the document
    • createDocument: create a new document
    • updateDocumentContent: update the content of a document
    • updateDocumentInfo: update auxiliary information of a document
    • getVariableInstance: return the variable instance object
    • getVariableValue: a convenient method that calls getParameterValue, and if it is a document reference, return the document content
    • setVariableValue: a convenient method that calls either setParameterValue or setParameterValueAsDocument, depending on whether the variable is a document variable
  • Environment information access methods:
    • getProcessId: return the process (definition) id
    • getActivityId: return the activity (definition) id
    • getActivityInstanceId: return the activity instance ID
    • getProcessInstanceId: return the process instance ID
    • getMasterRequestId: return the master request ID
    • getProcessInstanceOwner: return the owner type of the process instance
    • getProcessInstanceOwnerId: return the owner ID of the process instance
    • getActivityName: return the name of the activity
    • getProcessDefinition: return the process definition object (ProcessVO)
    • getParentProcessId: return the ID of parent process instance
    • getProcessInstance: return the process instance object (ProcessInstanceVO)
    • getPackage: get the current package object (PackageVO)
    • getEntryCode: get the completion code when the activity instance is created
    • getExternalEventInstanceDetails: for process instances that are started by external events only - return the contents of the external events
  • Script expression evaluation methods:
    • evaluateExpression: evaluate an expression in a number of supported languages
    • evaluateCondition: evaluate a condition (a boolean expression) in Magic Box language
    • executeScript:
  • Return code and message methods:
    • setReturnCode: set the return code, a.k.a. completion code, of the activity instance.
    • getReturnCode: get the return code, a.k.a. completion code.
    • setReturnMessage: set the return message (additional detail about the return code)
    • getReturnMessage: get the return message
  • Design time meta information:
    • getLable: return the label of the implementor
    • getIconName: return the icon name of the implementor
    • getDescriptor: return the pagelet descriptor of the implementor
  • Logging:
    • logwarn:
    • logdebug:
    • loginfo:
    • logsevere:
    • logexception:
    • logtag: return the tag used by all log methods
    • isLogDebugEnabled:

Implementing Adapter Activity Implementers

Although the framework is packaged with quite a few customizable adapter activities, it cannot be possible to cover all needs and hence you will likely need to define an adapter activity yourself. There are two primary ways to do this. The first is to extend an out-of-box adapter. Refer to the documentation of that adapter for how to do that. The second is to extend the base adapter activity class, AdapterActivityBase, which is an abstract class. Follow this link for the documentation for doing this.
