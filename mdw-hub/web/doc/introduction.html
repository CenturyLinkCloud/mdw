Introduction to MDW

What is MDW?

MDW stands for Model Driven Workflow. It's a CenturyLink-developed business process management system (BPMS) framework. A high-level summary of the BPM features offered by MDW is available here: http://cshare.ad.qintra.com/sites/mdw/bpm.

A business process is a defined set of business activities that represent the steps required to achieve a business objective. It includes the flow and use of information and resources. These are generally represented at a high level, so as to be understood by the users and managers of the process. Examples of business processes include:

  • Sales and ordering
  • Billing adjustments
  • Service activation
  • Order fulfillment
  • User access management

Business Process Management (BPM) is the concept of shepherding work items through a multi-step process. The items are identified and tracked as they move through each step, with designated users or applications handling the information.

Business Process Management System/Solution/Suite (BPMS) does not have a standard definition, but generally refers to tool sets that facilitate the following BPM activities:

  • Modeling – define current business processes and improvements needed
  • Execution – execute business processes and rules with varying degrees of automation
  • Monitoring – display real-time performance data and highlight jeopardy conditions
  • Analytics– identify process improvement opportunities and trends, with feedback to the model
  • Content Management — provide a system for storing and securing electronic documents
  • Collaboration — facilitate teamwork across organization boundaries such as Business and IT

Terminology: Processes, Activities, Transitions, Variables, Documents, and Tasks

A process in MDW is used to model a business process. Fundamentally it consists of a number of (automated or manual) work items or steps, called activities, which should be performed in a designated sequence. A process can invoke another process through a subprocess activity. Transitions are the links between activities that model dependencies between steps, so that when an activity is completed, the process flows through one or more outbound transitions to start other activities that depend on completion of the first. In more formal words: a process is a collection of activities and transitions between those activities, and is mathematically and graphically represented as a (typically acyclic) directed graph.

One of the benefits of a defined business process is that it can be applied repeatedly to many data instances. We use the terms process definition vs. process instance to differentiate between the template that describes the process and a particular execution of that process. Similarly, an activity instance is the execution of an activity within a process instance, and a transition instance is a transition actually traversed within a process instance. When the context is clear, we often simply use the term activity to mean activity definition or activity instance; similarly for the terms process and transition.

Along with process definitions, many other types of design-time artifacts are maintained in MDW. These include such things as Business Rules, Reports, Templates, etc., and are collectively known as workflow assets. MDW provides a distribution mechanism to encourage reuse of workflow assets. The MDW framework base distribution offers a number of commonly-used asset categories. Optional MDW extension packages include assets related to specific domains (such as Camel, Reporting, and a host of others). In addition, many workflow teams have contributed asset libraries that are discoverable through the standard MDW distribution mechanism.

Activities are implemented in MDW by Java classes, called activity implementors. To allow a single activity implementor to be reused in many places, it's typically designed to allow configuration through a set of attributes (name value pairs). For example, a web service adapter activity can take an endpoint URL as an attribute, allowing it to be used in multiple places where it connects to different service providers. In a workflow process, an activity definition therefore includes the specification of which activity implementor to use and the configuration of its attributes. The MDW framework makes it easy to extend the available set of implementors with application-specific ones, which are treated as first-class citizens alongside the prebuilt implementors. These in turn can be shared through the asset discovery mechanism described above.

For holding runtime data, a process is usually associated with a set of variables. A variable has a name and will be give a specific value at runtime. The value of a variable is constrained to a given type declared when designing the process. A design-time variable representation is more formally called a variable definition which defines a name and type but not a value, whereas its runtime instance is called a variable instance, which holds the value corresponding to a specific owning process instance.

Documents are special types of variables that can hold large data values such as XML documents or Java objects. These are passed by reference to avoid keeping multiple copies.

A task is an activity meant to be performed by a person. To avoid confusion we often use the term manual task since task is used in some BPM products to express the more general concept that we call activity (both manual and automated) in MDW.

A task manager is a (typically web-based) user interface enabling users to perform many functions related to manual tasks. Examples include:

  • Reviewing tasks currently assigned to the user
  • Searching tasks using various criteria
  • Viewing and editing details of a task
  • Claiming, completing, canceling, forwarding tasks
  • Managing workload by distributing tasks among users
  • Managing users, groups and roles

In MDW the task management UI is known as MDWHub. Tasks relate back to the process definition through manual task activities. The definition of a task is called a task template. This represents design-time aspects (such as Due Date and Notification recipients), whereas a task instance is a runtime line item in MDWHub that contains specific data and is assigned to a designated user for completion. The data captured in MDWHub feeds back to the process instance, and can be used for runtime branching and decision making. A special type of task may exist without reference to any workflow process. This type of task is called an ad hoc task, which will be covered later.

MDW Components

From a code organization perspective, MDW provides a framework that can be customized through configuration and scripts as well as extended with custom activities and event handlers, which are implemented in Java. The following diagram shows the major components of MDW and workflow applications built on top of MDW.

The MDW Framework contains the following major components:

  • Designer (available as an Eclipse plug-in and also as a standalone application called Designer RCP).
  • Workflow Engine, the module for executing business processes.
  • Workflow Implementation, the library of available workflow assets (such as activity implementors).
  • MDWHub, the task management web application
  • Services Framework, the extensible SOA platform
  • BAM and Reporting, for runtime monitoring and custom reports

MDW (and workflow applications built on MDW) can run on a number of supported server containers, and use an Oracle or MySQL database for storing both runtime data. The core of MDW uses the servlet API of the container and can optionally take advantage of other container provided features such as JMS, transaction management, database connection pools, thread pools, OSGi, clustering, e-mail, etc. Specialized MDW adapters and listeners for licensed, third-party protocols such as Tibco Bus and MQSeries depend on corresponding server-side installations (but these are required only when an adapter or listener is actually utilized by a workflow app).

Leveraging MDW to Build Workflow Applications

MDW supports a number of different extensibility models:

  • Cloud Model: Deploy into the standard CenturyLink internal Cloud infrastructure. Typically this involves installing the MDW war on Tomcat, with a MySQL database for storing runtime data. Applications deploy their source code through Dynamic Java or custom Jar assets. Extensibility points are managed by way of source code annotations. Running MDW in cloud mode allows a workflow app to take advantage of the well-publicized benefits of running in the cloud (such as on-demand capacity, self-service provisioning, elastic computing power, etc).
  • OSGi Model: MDW can be deployed on Apache ServiceMix or JBoss Fuse. In this mode a workflow app's source code is delivered in the form of OSGi bundles, and extensibility touchpoints are provided through the OSGi service registry. Certain types of workflow apps may benefit from OSGi features, especially those that include long-running processes which must support in-flight compatibility for previous releases.
  • Hosted Model: Because the CenturyLink corporate repository contains a wide array of prebuilt MDW workflow assets, some applications may be able to build an entire workflow without any custom Jar at all. Business logic is implemented using Groovy, JavaScript, Dynamic Java and Drools. Applications in this mode share a common database, relying on the MDW isolation and customization features to keep individual apps logically separate.

Requirement Gathering and High Level Design

MDW, being a BPMS system, preaches a top-down design philosophy. The designer should be used as the tool to facilitate requirement gathering and communication between clients, system engineers, analysts, and development leads. The development will continue by enriching the information in the designer, resulting design and implementation.

Therefore, during process design stage, the participants will be clients, system engineers, analysts and development leads, and the output will be high level processes with documentation of activities (but without implementation details). At this stage, we can use any activity as place holder (you probably want to avoid using activity with obvious special meanings such as start and stop activities) and simply document the requirement for the activities. Right-click on any activity and select "Document" menu item (or simply press Control-D while the activity is selected) to open up documentation dialog for the activity.

For interfaces with external systems, the documentation dialog allows to specify hyperlinks, which should be used to point to interface specification documents, such as XSDs.

For manual tasks, we can use GUI builder to create mock up screens. To be implemented: when printing out requirements, print out screen as well.

Use embedded processes for exception handling, time out handling, request modification and cancellation handling.

Low Level Design

Once the requirements are agreed upon, we can start to move down to next stage on low level design. The participants at this stage is typically the entire development team, driven by the development lead. Low level design involves typically the following steps:

  • Enrich requirements for activity to include more details
  • Define variables associated with processes and document their intended usage.
  • Determine which activities can use out-of-box implementors and which activities must use new activity implementers.
  • Determine if custom event handlers are needed.
  • Assign activity implementors and event handlers to developers
  • Configure processes, activities, event handlers
  • Configure or develop resources

More on Processes and Activities

Different Ways for Starting Processes

A process is typically started by one of the following means:

  • Upon receiving an external event
  • Upon submission of a form that specifies to start a process.
  • Invoked by another process. Processes started this way are called subprocesses, and the process starting them are called parent processes.
  • Started by the MDW scheduler (a.k.a timer)
  • Will be supported in future: as a result of rules firing by a rules engine.

Service vs. Regular Processes

Business processes are often long running, lasting days or months. Therefore, if a process is started in response to an external event (a message from an external system), the external system does not wait for the process to complete and at most obtains an acknowledgment that the process is indeed started. In another word, processes are typically started asynchronously.

MDW processes can also be used to implemented services, in which case the external system is waiting for a response in real time. Such processes are executed synchronously, and are called service processes. We will call processes executed asynchronously regular processes. Unlike regular processes, service processes typically have strict performance requirement, as the external system is waiting for responses. Because response time may be crucial for service processes, MDW 5.1 currently supports 10 different performance levels. Earlier versions of MDW support three levels. See Processes for details.

Embedded Processes

There is another type of processes, called embedded processes, which are embedded in regular processes. Embedded processes are used to model handling of exceptional events, and have the following types:

  • Fallout handlers: for handling error conditions while executing activities
  • Cancellation handlers: for handling cancellation of a request during exection
  • Correction handlers: for handling changes to a request during execution
  • Delay handlers: for handling timeout conditions while an activity in waiting status is timed out (SLA expires)

Remote Processes and Federated Workflows

MDW also supports remote process invocation and hence federated workflows. A process in an MDW application can directly invoke another process hosted in a different MDW application as a remote subprocess. The designer can display remote process instances in pretty much the same way as local process instances, thereby allowing the user to drill down from the main process to all subprocesses (remote or local) without swiveling to different designers. The main constraint is that the designer cannot make changes to remote processes or process instances (anything from remote application can be accessed read-only)

Parameter passing is also done pretty much the same way as local subprocess invocation. Document variables are passed down as references, but they are read-only as well.

Developing Activity Implementors

In general, developing an activity implementor involves two steps: creating a java implementor class (most often by extending an out-of-box implementor), and making it available in the designer. Refer to Introduction on Developing Activity Implementor for more detailed description on this topic.

Application implemented activity implementors should also be implemented in such a way that it allows configuration and comes with documentation, to allow reusability.

Adaptor and Connection Pool Framework

An adapter is an activity used to send messages to external systems, and receive responses if the communication protocol is synchronous. If the business logic expects an asynchronous response for a one-way message sent to external systems, the listener framework to be introduced later will be used to listen to the responses (as external events) and notify an event wait activity waiting for the responses.

The above out-of-box activity list contains quite a few generic adaptors for common transport protocols, including JMS adaptor, Tibco bus adaptor, MDW web service adaptor, Axis based web service adaptor, RESTful web service adaptor, MQ Series adaptor, LDAP adaptor, etc. It is typical for applications to extend these generic adapters to implement external system specific adapters.

These protocol specific out-of-box adaptor activities are very simple to use. This simplicity, while allowing rapid learning and prototype, may lack manageability needed in a production environment. The connection pool mechanism, while slighly more complicated, adds the following features for supporting production environments:

  • Throttling control: connection pools allow to specify maximum number of simultaneous connections.
  • Certified delivery: connection pools can automatically re-send messages on connection failures. You can configure how many and how often the retries are to be made. When maximum number of retries are reached, an exception is thrown back to the invoker when it is synchronous, or a Sentry marker is logged to generate alarms when it is asynchronous.
  • A management user interface is provided to manually re-send messages after all automatic retries fail, among other management functionalities.
  • Switching to a different transport technology is easier (configuration change that can even be done in the management user interface at run time).

There is a single out-of-box activity, connection pool adapter, that can be used directly or extended to send messages through connection pools. We call those adapters connecting directly to external systems direct connection adapters. Connection pools use poolable adapters, which are adapter activities implemented in a new way introduced in MDW 4.5.12 and MDW 5.1.05. Refer to Using Adapter Connection Pool and Using Adapter Activities for information on how to configure connection pools and on how to implement (poolable) adapters.

Implement Adapter Activity Implementors

One of the most common reasons for implementing new activities is to define adapters specific to some external systems. Refer to Using Adapter Activities for more detailed description on this topic. You may also need to refer to the documentation for the specific type of adapter you are to extend.

Certified Messages

Although connection pools and direct connection adapters can be used to automatically re-send messages on conneciton failure, JMS is the preferred way for implementing certified messages, which not only garantees the messages are delivered but also ensures the messages are delivered no more than once. Introduction to certified messages is covered under JMS Adapter documentation. The certified messages through JMS, however, require both the sender and the receiver are MDW applications (4.5 for MDW 4 and 5.1 for MDW 5), because they require hand-shaking between the sender and receiver to ensure the messages are acknowledged and delivered exactly once.

When the receiver is not an MDW application, it is not possible to implement a general certified mechanism. With the help of a little bit of custom coding (in adapters), the connection pools can be used for the next best thing - it can ensure the messages are delivered by automatic retrying, but it cannot ensure the messages are delivered no more than once. In practice, this is sufficient in most of time, especially that it does not require receiver to be MDW-based applications.

We note that the certified messages addressed above refer to MDW implementation on top of underlying transport protocols. Some transport protocols support certified messages natively, and when you use those features, you do not need to use MDW implementation. However, we found in most cases the applications are not able to use native certified messages for various reasons:

  • MQ Series has a very solid implementation of certified messages, but the company (on legacy Qwest side) discourages the use of MQ Series.
  • Tibco BUS certified messaging has a lot of technical issues that the bus infrastructure team strongly discourage its use
  • WebLogic JMS messaging by default does not garantee delivery. There are options to configure that way (called store-and-forward), but setting that up turned out to be really challenging, and there are severe limitations where they can be used

Listener and External Event Handler Framework

Messages received from external systems (other than those as synchronous responses to messages sent to external systems) are called external events. We note that MDW engine uses messages internally to execute processes, and these messages are called internal events.

The entities receiving external events directly are called listeners. Because the transport protocols can vary, and the message formats are typically different, listeners in MDW typically performs the following tasks:

  1. Extract the text contents from the protocol specific message objects.
  2. Determine a external event handlers (often abbreviated as event handlers) to handle the messages, using a common MDW Java object called listener helper. The listener helper makes the decision based on text contents and user configurations (commonly by the root elements of XML messages), but independent of transport-protocol.
  3. Persist the message contents in the database (in DOCUMENT table)
  4. Delegate the processing to the event handlers, which may return responses/acknowledgement
  5. Send responses or acknowledgment returned by the event handlers back to the external systems.

Listeners are the only protocol-specific code components, and they perform minimum amount of work and often contains less than 100 lines of code. Event handlers and the listener helper are all protocol agonostic. MDW provides listeners for many common protocols out-of-box, such as JMS, Tibco Bus, RESTful webservice, MQ Series, MDW SOAP web service, EJB. MDW applications rarely need to define their own listeners.

The listener framework assumes the external events are all XMLs. Through MDW Designers, the applications use XPath to configure which external event handlers to handle the messages (root element tag is just a special case for XPath).

An external event handler is a Java class, and it may may perform one of the following main functions:

  • Start a process instance
  • Send a message to a waiting process instance (solicited or unsolicited such as cancel and pending order changes)
  • Invoke a synchronous process and return its response
  • Respond directly with all custom code and no touch to processes.

Manual Tasks

Pre-flow vs In-flow Tasks

We already mentioned that MDW task manager also supports tasks that are not related to activities, independent from processes, and such tasks are called independent tasks. The most common usage of independent tasks is to start a process. Such tasks are also called pre-flow tasks, as they exist before the corresponding process instances are created, and the process instances are created only at the completion of the pre-flow tasks. In contrast and when it needs to be emphasized, we call tasks associated with task activities in-flow tasks.

Autoform vs. Custom Tasks

Being the user interface components, tasks have model, view and control aspects. MDW uses Java for embedding dynamic values.

Configure Task Activities

Task activities are configured in the designer, just like any other activities. Follow this link for details on configuring autoform manual tasks, and follow this link for details on configuring custom manual tasks.

In summary, you will need to configure the following items:

  • Task names and descriptions - these are displayed in task manager
  • The groups to which the tasks are to be associated
  • Task due date (a.k.a. SLA)
  • Forms to use for general tasks, and variable to display for autoform tasks
  • Email notification recepients and templates, if needed
  • How to handle unsolicited external events while waiting for users to complete the tasks
  • Auto-assignment algorithms to use if needed
  • Jeopardy alert intervals.

MDWHub

MDWHub is the web front-end for the users and administrators to perform and manage manual tasks. It supports common functionalities such as task searching, task summary view (list of tasks based as the search result), task detail view (view details of a single task), user and group management, quick links (most commonly used queries such as "My Tasks"). The MDWHub User Guide describes end-user functionality, whereas MDWHub Development discusses customization techniques.

MDWHub uses ClearTrust single-sign-on based on LDAP credentials.

For authorization control, MDW employs the concepts groups and roles. The following highlight the scheme:

  • A user can be a member of one or more groups.
  • A user can have roles associated for each group he/she is a member. If no roles are configured, the user can only view resources associated with the group but cannot make any changes or take any actions.
  • Out-of-box roles include:
    • Process Design: can create and modify process definitions and other workflow assets owned by the group.
    • Process Execution: can execute processes and manage/alter process instances of process definitions owned by the group
    • Task Execution: can take actions on manual tasks that are associated with the group.
    • User Admin: can add/delete users to the group and grant roles.
    • Supervisor: can act on behalf of other members of the group; can assign/reassign tasks when other members are not allowed.
  • Custom roles can be added as well (e.g. for the purpose of restricting what actions a user can take on manual tasks).
  • A manual task can be associated with one or more groups, and a user can see the task as long as he/she is a member in at least one group that the task is associated with.
  • A process or any other workflow asset (to be covered later in this document) is owned by one group. Only users who are members of the group can see it.
  • Groups can have hierarchy - a group can be designated as s parent group of another. A user with a role in the parent group will automatically have the role in the child and descendent groups.
  • Site Admin is a special group that is the parent of all other groups without specifically designated parent groups. Hence any user who is a member of this group will have the roles defined for this group carried over to all other groups.
  • There is also a Common group, of which any user is a member. Process definitions and other workflow assets owned by this group can therefore be seen by all users.

Customizing MDW Hub

We have introduced above different ways to customize detail views and actions of a task. It is also commonly needed to customize common functionalities of the MDWHub such as summary view, user and group management, queries and quick links, etc. MDW provides several ways to support such customization:

Remote and Foreign Task Managers

Starting from MDW 5.1, the MDW task manager is separated from the workflow engine, so that multiple workflow applications can connect to the same task manager that is hosted by another (may be dedicated) MDW application. This is to allow users to access one task manager for any workflow tasks, and to allow managing users, groups and roles in a centralized location. If an MDW application uses a centralized task manager (or any MDW task manager not hosted in the same application), we will refer to such task managers as remote task manager. When it needs to be emphased, we call the MDW task manager coupled with the engine in the same application the local task manager.

Because MDW engines communicate with task managers through messages, using the API introduced here, they can also communicate with non-MDW task managers as long as as long as they implement the same task manager API. We will call such task managers as foreign task managers.

Because detail view of task instances and actions to be performed on these tasks are closely related to the workflow application, when a remote task manager is used, MDW allows to display task detail views still in local task managers, and the remote task manager displays the task summary views and link seamlessly to the detail views hosted by individual applications.

Two properties need to be set for the engine to use centralized or foreign task manager:

  • mdw.taskmanager.jndi - this is the JNDI for the task manager server JNDI provider, and it needs to be set in the form t3://host:port. We have not tested non-WebLogic provided JNDI servers as of now.
  • mdw.taskmanager.queuename - this is the JMS queue name of the task manager. If you do not specify this property, the engine will use com.centurylink.mdw.external.event.queue, which is the queue name for MDW-provided task manager.

The communication between MDW engines and task managers is through standard external event infrastructure MDW supports, namely it can be any common protocol such as JMS, WebService or Tibco Bus. The preferred is JMS, which is used when the task manager is MDW-implemented.

Scheduled Jobs

Scheduled jobs are jobs to be run at scheduled times, typically repeating periodically. A typical usage is to start a process periodically (e.g. at the same time every day), and another is to run MDW database clean-up script periodically.

System Configuration, Build and Deployment

Properties and configuration

The property manager loads properties in the following sequence (if a property is defined in more than one place, the one loaded latest overrides the previous one, effectively making the later loaded property file the higher precedence):

  • mdw.properties in the application's configuration directory (this is typically the Qwest/config directory under the domain, or deploy/config in the application development directory for cloud development environment). This file contains properties used by MDW components and is distributed along with MDW application. The application should not typically modify this file.
  • application.properties in the application's configuration directory. This file should contain application specific (non-MDW) properties, as well as MDW properties that override those in mdw.properties. MDW distribution does not include this file, and it should be created by the application when there is a need.
  • MDW Database table ATTRIBUTE, where ATTRIBUTE_OWNER='SYSTEM' (ATTRIBUTE_OWNER_ID can be anything), or 'PACKAGE' (ATTRIBUTE_OWNER_ID must be the package ID the current process instance is associated with).
  • env.properties file in the application's configuration directory. This is intended for use in development environment only, to provide local override.

Prior to MDW version 5.1.08, ApplicationProperties.xml (which is in XML format rather than standard Java property file format) was used in place of mdw.properties and application.properties files.

Server Clustering for Load Balancing and Fail Over

MDW Framework supports server clustering for load balancing and fail over, most often through the underlying servlet containers. Because MDW and its applications are mainly workflows rather than web applications, the clustering needs are different - instead of or in addition to http proxies that are used for clustering web sessions, MDW requires load balancing and fail over for internal and external messages. Indeed, it is recommended not to use session replication feature of HTTP proxies for MDW applications. The following table lists the key features and how they are implemented and configured for the containers.

  • Server name: each clustered server must have a unique name. WebLogic servers already have names. For Tomcat, the name for each server should be specified in MDW property mdw.server.this.
  • Internal messaging through JMS. For WebLogic, distributed queues are used for clustering purpose. Tomcat does not support JMS natively, so this is not applicable. If ActiveMQ is used with Tomcat, it supports distributed queues as well.
  • Internal messaging through RMI. MDW natively implements a round-robbing algorithm to support load balancing, but fail over is not supported. The list of servers is specified through the property mdw.server.list.
  • Internal messaging through RESTful web service. For Tomcat, it utilizes the HTTP proxy, which should be specified by the property mdw.proxy.server. If no proxy is configured, MDW uses the internal round-robbing algorithm just like the one for RMI internal messaging. Similary, the proxy for WebLogic servers is used for clustering, although with the current deployment model in ECOM, we will need to make sure the internal messaging URL is excluded from clear trust interception (for single-sign-on).
  • Broadcasting to all servers. If JMS is used for internal messaging, the broadcast is supported by underlying JMS provider. If RMI or RESTful web service is used for internal messaging, MDW sends the message to every server specified in the property mdw.server.list.
  • Connect to a specific server. There are cases where we want to connect to or send message to a specific server, such as the file panel application. For WebLogic, this is currently only possible through RESTful web service. For Tomcat, RMI can also be used.

How to read log files

MDW implements a simple logger that uses stand output for logging. The logging information has proven to be really valuable for trouble shooting. Here are sample log entries: 
[(i)20091119.10:14:58.349 p358911.463591 a358921.585796] Activity started - Process Start
[(i)20091119.10:14:58.443 p358911.463591 a358921.585796] Activity completed - completion code null
[(i)20091119.10:14:58.959 p358911.463591 a358923.585797] Activity started - Calculate sqaure
[(i)20091119.10:15:01.804 p358911.463591 a358923.585797] Activity completed - completion code null
[(i)20091119.10:15:02.304 p358911.463591 a358915.585798] Activity started - Wait for a while
Thg log tag (information contained within the square brackets) contains the following information:
  • Message log level: (i) for informational, (d) for debug, (w) for warning, (s) for severe.
  • Date and time when the line is logged
  • "p" followed by process definition ID, dot, and process instance ID
  • "a" followed by activity definition ID, dot, and activity instance ID

Applications can use the same logging mechanism inside an implementer - two new methods are added to the base class: loginfo and logdebug. For example, super.loginfo("I love this log entry") will generate the following log line: 

[(i)20091119.10:15:02.304 p358911.463591 a358915.585798] I love this log entry

The logging level is controlled by the MDW application property mdw.logging.level. It takes value value "WARN", "INFO", "DEBUG", and "MDW_DEBUG", in sequence of more logging information. "INFO" is the default.

Order Management using MDW

Order life cycle management is a broad area where business processes are extensively involved. It is therefore not surprising that more than half of all installed MDW applications fall in this area. We devote this section introducing general architecture for order management using MDW and MDW-based applications. The following figure depicts this architecture at a high level.

The creation of orders start in order entry systems such as Consulting Plus, Ensemble and IOE. These systems may invoke many services to collect information for order creation, such as:

  • Address validation
  • Facility check
  • Loop qualification
  • Credit check
  • Appoint scheduler
  • TN and facility reservation

Implementation of these services may involve accessing multiple source systems, and any reservation may result needs for un-reservation. The service processes of MDW provide desired functionality as well as tunable performance levels to balance response time requirement with traceablity. As MDW is relatively new in this area, it is currently only used in some nitch areas for pre-order services.

Once an order content is completed, it should be submitted to workflow systems to manage its entire life cycle until the order is fulfilled (provisioning is completed) and the billing system is updated. All orders should (but not yet) be submitted to Integrated Order Management (IOM) workflow, which is an MDW based workflow and responsible for decomposing the order into product-specific components and invoking corresponding subworkflows. We sometime refer to IOM as the macro workflow, whereas product specific workflows as micro workflows, although there is really no reason to limit the hierarchy to just two levels. Besides top level order decomposition, IOM provides orchestration between micro workflows. For example, IOM may need to ensure billing process not started until all products are successfully provisioned; it may need to start layer 2/3 provisioning processes only after layer 1 provisioning processes have been completed.

Unlike some commercial systems such as OSM, MDW or IOM do not provide out-of-box order decomposition functionality. Instead, MDW/IOM relies on an (any) external rules engine to decompose an order into an execution plan (which is an XML file describing what subprocesses need to be launched), and MDW has an out-of-box activity that can launch multiple heterogeneous subprocesses based on the execution plan. Even though out-of-box order decomposer such as the one in OSM does well as showcase, MDW architecture is simple yet flexible and has the following advantages:

  • The open architecture allows to plug-in many external order decomposers, and CenturyLink happens to have many such existing ones (ProductBuilder, FTS, etc).
  • The architecture can work with any order format, whereas OSM order decomposer can only work with orders defined in OSM formats; translation from existing order formats to other formats are typically probititively expensive.
  • It is often convenient to have multiple levels of order decomposition. For example, IOM decomposes into Ethernet and VoIP order, then Ethernet micro workflow decomposes the Ethernet product further into UNI and EVC tasks. MDW architecture does not restrict the order decomposition as a single step, as the different order decomposers can be consulted at any time, and heterogeneous subprocess launching activity can be used more than once (there is no distinction of "orchestrated" processes from other processes).

Another important component for order management is Order Services Repository (OSR). In an ideal world, we should have just one database for storing service orders. For historical reasons, CenturyLink has several major order databases (SOPs, PROD, Ensemble and IABS) with many minor ones. Most of these are not easy to access and are certainly not designed to work with a BPM system like MDW. OSR is designed for mainly two major purposes:

  • For buffering orders stored in major order databases so that workflows (regardless whether they are MDW-based) can easily access. It also allows associating temporary data collected and used by workflows during order life cycle (such as circuit IDs, device CLLIs, and intermediate critical dates).
  • For consolidating service orders stored not in major order databases but in many small, product specific databases (such as VoIP orders, DTV orders, Verizon wireless orders, network planning requests, etc).

Detailed introduction to OSR can be found in Service Order Repository - High Level Design

BAM (Business Activity Monitor) that we introduced above is also an important component in order management architecture, as it provides a central place for tracking orders from end-to-end.

Handling supplemental orders (cancelling or modifying orders after they are submitted but before they are completed) is an important task of any order management systems and typically the most complex one. MDW provides support in the following ways to assist supplemental order handling:

  • Two internal event types are dedicated for handling receiving order cancellation and modification: ABORT for cancellation and CORRECT for modification.
  • Any long running activity (an activity that can get into wait status, such as manual task activity, timer wait activity and event wait activity) can be configured to receive unsolicited supplemental order messages.
  • Two embedded subprocess types, Cancellation and Correction handlers, can be used to handle order compensation processes. Activities receiving supplemental order messages can rasie ABORT and CORRECT events to start the embedded processes.

Custom Property Manager

You can override the MDW property manager by your own. First, you need to create y our property manager, which must be a java class implementing the interface com.centurylink.mdw.common.utilities.property.PropertyManager. You can use the default MDW property manager com.centurylink.mdw.common.utilities.property.impl.PropertyManagerDatabase as a sample, or better subclass from it, as you may need to use the methods in that class. Secondly, you will need to direct your application to use this property manager instead of the default one. This is done through Java system property property_manager. For example, you can set the property in the command line of starting weblogic as

 	 -Dproperty_manager=com.qwest.oas.OasMdwPropertyManager
